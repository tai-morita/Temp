Ⅰ．目的
 プログラムの標準的なコーディングスタイルを規定することにより、無秩序なプログラミングを防止し、保守性の高い、さらには、移植性の高いプログラムの実現を目的とする。 一般的には、よりわかりやすいコーディング方法がより良い方法である。1行1文字少ないことよりも、3年後に読み直した時にすぐに思い出せるのか、の方が重要である。

Ⅱ．変数名
１．	無理に短い名前をつけようとせず、多少長くなっても内容説明的な良い名前をつけるようにする。
２．	似たような名前は避ける。一見して区別できるような名前が有効である。
３．	変数名の最初にはその変数の型を示す文字を小文字でつける。
［例１］vector<int*> vecpiVerticalTableCenter;
      vec：vector標準テンプレートであることを示す。
      ｐ ：ポインタであることを示す。
      ｉ ：ｉｎｔ型であることを示す。
［例２］int aiLine[NUMBER_OF_PIXEL];
      ａ：実体をもった配列（array）であることを示す。
      ｉ：ｉｎｔ型であることを示す。
４．	型を示す小文字の後には、大文字で始まる内容を示す英名を付ける。日本語名をローマ字で記述することは避ける。
５．	メンバ変数の最初には”m_”をつける。
 ［例］int m_iX;			// 整数の X 座標
６．	ｆｏｒ文などのループを制御するイテレータは、型とイテレートする意味とする。
　[例]iX　int型で画像の水平方向の画素を走査するイテレータ
７．	長い変数名を記述する際(複数の単語で変数名を記述する場合)は、1単語の先頭文字を大文字で表現するようにする。
  [例]　　メソッド1をコントロールするフラグとして、下記変数名で記述する。
　　　　変数名：dwMethod1CtrlFlag
　　　　　　　　dwは変数がDWORD型であることを示す。
　　　　　　　　又、Method1、Ctrl、Flagと1単語単位で、先頭文字を大文字とする。
８．グローバル変数はなるべく使用しない。使用する場合は、必ず、最初に”g_”をつける。
    グローバル変数の宣言は、ヘッダファイル.Hで宣言する様にする。
  [例]　　　変数名　g_dwMethod1CtrlFlag
                 上記変数は、DWORD型のグローバル変数であることを示す。

Ⅲ．関数名
 １．関数の名前は、
“大文字で始まる具体的な動詞”＋“大文字で始まる具体的な目的語”
     で表現するようにする。従ってこのためには、関数の機能自体が上記の名前で表現されるように設計されている必要がある。
［例］// Voxelsからスライス画像をつくる。
      void CCt::MakeSlices(void)

Ⅳ．クラス名
 １．クラス名は、必ずCで始まる様に記述すること。
     変数名同様、 長いクラス名(複数の単語でクラス名を記述する場合)を記述する際は、1単語の先頭文字を大文字で表現するようにする。
  [例]　3DXで、画像処理を実行するクラスとして、下記クラス名で記述する。
　　　クラス名；CDxImageProc
 ２．ダイアログのクラス名は下記のように最後に”Dlg”をつける。
 ［例］class CTestCameraDlg : public CDialog {

Ⅴ．定数
 １．#ｄｅｆｉｎｅによる定義
     数値が直接書かれている定数は、数値の持つ意味が分からないうえに、プログラム修正の際に探し出すのが大変である。
     また、“定数値＋１”、“定数値－１”のように派生してくるデータは、厳密にロジックを追わない限り修正は不可能である。従って、数値定数は意味がわかるような名前をつけて＃ｄｅｆｉｎｅで定義する。これにより、定数の変更時も修正は１箇所ですむ。
     ただし、ポインタ値や添え字に１を加えるとか１を減らすような場合は数値のままがよい。
     定数名はすべて大文字とし、単語の区切りは＿とする。
［例］// the buffer size is calculated to be about 1/20 sec
      #define UPDATES_PER_SEC     (20)
 ２．const 変数による定義
     #define による定義のスコープはグローバルなので思わぬ副作用をもたらす。
     クラス内で必要な定数は const なメンバ変数を用いることを推奨する.
［良い例］
      class CBuffer {
          const size_t m_kuiBufferLen;
          CBuffer():m_kuiBufferLen(20){}
      };
［悪い例］
      #define BUFFER_LEN    (20)
      class CBuffer {
          void* m_pvBuffer;
          CBuffer() {
              m_pvBuffer = malloc(BUFFER_LEN);
          }
      };

 ３．ｓｉｚｅｏｆ演算子の利用
     使用しているマシンのデータサイズを直接用いてはならない。
     プログラム中で変数の占めるバイト数を用いる場合はｓｉｚｅｏｆ演算子を用いて表す。

 ４．文字列の定義
     文字列を宣言する際、＃ｄｅｆｉｎｅで文字数を設定する。従って、文字列を格納する配列の宣言は、NULL文字を考慮して、
      #define CHAR_LEN 10
      char pcText[CHAR_LEN+1];
     という形となる。

Ⅵ．変数の局所化
１．	メンバ変数はなるべくｐｒｉｖａｔｅあるいはｐｒｏｔｅｃｔｅｄとする。クラス外でメンバ変数を操作する必要がある場合は、なるべく関数を介して操作するようにする。
２．	ｆｏｒ文のイテレータは下記のようにコーディングし局所化する。
［良い例］
		int iPositionInLine = 0;
		bool bFound = false;
		for(int iIndex = 0; iIndex < sizeof(aiLine)/sizeof(aiLine[0]); iIndex++){
			if(aiLine[iIndex] != 0){
				iPositionInLine = iIndex; bFound = true;
				break;
			}
		}
		if(bFound){
			printf("found index = %d\n", iPositionInLine);
		}else{
			printf("not found\n");
		}
［悪い例］
		for(int i = 0;i < NUMBER;i++){
			if(iLine[i] != 0){
				break;
			}
		}
		printf("found index = %d\n",i)



Ⅶ．関数
１．	関数の機能
　関数に持たせる機能は、以下の形式で要約できるように設計することが望ましい。
　 “具体的な動詞＋具体的な目的語”
　長いコードの一部分を抜き出しただけのような関数は、プログラムの構造をわかりにくくし、不適切である。
２．	関数の局所化
  クラス内でしか呼び出されない関数は、ｐｒｉｖａｔｅ宣言し、外部から隠蔽する。
３．	評価の順序があいまいな関数の引数にインクリメント演算子は使わない。
［悪い例］
		printf("%d:%d\n",++i,++i);
４．	関数の大きさ
関数は論理のまとまりとなるようにし、なるべく短くする。行数は２００行以下が望ましい。

Ⅷ．マクロ
１．	マクロ名
マクロ名はすべて大文字とし、単語の区切りは＿とする。
  ［例］#define AREA_OF_CIRCLE(x) (PI*(x)*(x))
２．	引数付きマクロ
マクロ拡張全体をかっこでくくる。マクロ展開によって式の意味が変化することを防ぐためである。
 ３．マクロの副作用を発生させるようなコーディングはしない。
　［悪い例］
　#define SQURE(n) (n*n)
　main(){
　	int i = 1;
　	while(i <= 10)｛
         printf("%d\n",Square(i++));
	｝
  }
は、次のように解釈される。
main(){
	int i = 1;
	while(i <= 10)｛
         printf("%d\n",Square(i++*i++));
	｝
}

Ⅸ．使用禁止
１．	goto文はなるべく使用しない
goto文を使用すると、エラーが発生しやすいし、読みにくいコードになりやすい。Goto文は下記のような場合以外では使用すべきではない。
・２重以上のループから抜ける処理
２．	１文字の変数は使用すべきではない。
１文字の変数は、変数の意味がわからない上に、プログラムの修正の際に検索不可能である。

Ⅹ．スタイル
１．	クラス定義記述順序
機能ごとにまとめて記述する。


２．	ｉｆ文
ｔｈｅｎ部、ｅｌｓｅ部が１行の場合でも中かっこは省略しない。
［例］
		if(bFound){
			printf("found index = %d\n", iPositionInLine);
		}else{
			printf("not found\n");
		}
３．	ｆｏｒ文
    ｆｏｒ文のイテレータを局所化するため、下記のようにコーディングする。
 ［例］
		for(int iIndex = 0; iIndex < NUMBER_OF_PIXEL; iIndex++){
			if(piLine[iIndex] != 0){
				iPositionInLine = iIndex; bFound = true;
				break;
			}
		}
　このようにコーディングすることにより、処理単位での独立性が高まり、結果として可搬性、可読性が高まる。
　[例]
		for(int iIndex = 0; iIndex < NUMBER_OF_PIXEL; iIndex++){
　　　　処理Ａ
		}
		for(int iIndex = 0; iIndex < NUMBER_OF_PIXEL; iIndex++){
　　　　処理Ｂ
		}
　旧規約では、イテレータの局所化のためｆｏｒ文自体を {} で囲うことになっていたが、C++ の規約上不要なので、必須としないこととする。

４．	ｓｗｉｔｃｈ文
“break”と”case”を必ず対にして1行に記述すること。
　［悪い例］
　  switch(expression){
        case A:
            statementA;
   	    break;
        case B:
            statementB;     ←breakがない。
        case C:
            statementC;
	    break;
       default:
            statementD;
    }
　
　［良い例］
　  switch(expression){
    	{// None
　　　　}break; case A:{
           statementA;
　　　　}break; case B:{
           statementB;
           statementC;
　　　　}break; case C:{
           statementC;
　　　　}break; default:{
           statementD;
	}
    }
［悪い例］では”case B”に対して”break”がないが、コーディングミスで”break”がないのか、意図的に”break”を記述してないのかが、わからない。

ⅩⅠ．構造体
１．	構造体には、privateな変数を持たせない。
２．	構造体には、関数を持たせない。
３．	上記が必要な場合は、クラスとする。
 ４．構造体宣言は、ヘッダファイル.Hで宣言するようにする。

[例]
         template <class T_Point>

ⅩⅢ．列挙型
１．	識別子名は、１文字目を大文字の”E”とすし、変数名同様、長い識別子名(複数の単語で識別子名を記述する場合)を記述する際は、1単語の先頭文字を大文字で表現するようにする。
２．	列挙子名は、先頭に”E_”をつけ、すべて大文字とする。
３．	変数名は、先頭に小文字で識別子名をつける。
［例］
　　　　enum EPanoramaMode{
　　　　	E_NORMAL,
　　　　	E_CROSS,
　　　　	E_PAL,
　　　　};
　　　　EPanoramaMode epanoramamodeAChannel;
　　　　EPanoramaMode epanoramamodeBChannel;
　　　　
ⅩⅣ．コメント
１．	簡潔なコメント
プログラムには注釈を書くべきであるが、むやみにたくさん書けば良いというものではない。簡潔な注釈で良いから、適所に的確に書くことが重要である。プログラムの読みやすさは、注釈の豊富さによるのではなく、文の構造の明確化と意味のある変数名に負うところが大きい。
 ２．各クラスのヘッダーファイルの先頭には、目的、前提条件、注意事項を記述すること。
